# termd 0.6.0

## 用户可见变化

- 破坏性升级：relay 信任模型从“不可信 relay + 运行时 E2EE”切换为“可信 relay + daemon 注册”。线上链路更短，pairing/auth/session/file 权限仍由 `termd` 最终校验，但 relay 现在是受信入口，不再按旧 E2EE 威胁模型设计。
- `termrelay` 新增 setup token 和 daemon registry。relay 安装时生成 `/etc/termd/termrelay_setup_token`，daemon 使用 setup token 注册，relay 持久保存 `server_id -> daemon token hash` 到 `/var/lib/termrelay/daemon-registry.json`。
- `termd` 连接 relay 时自动生成并持久化 daemon token，用户不需要手工管理 daemon token。安装脚本通过 `--relay-setup-token-file` 完成 daemon 注册。
- 浏览器和 `termctl` 经 relay 配对时改用短期 pair-ticket admission；`termd pair --qr` 会先把 pairing ticket 注册到 relay，再输出 `termd-pair:v2:...` 邀请码，避免拿到一个 relay 无法使用的假成功 invite。
- `termctl` 配对路径区分“relay route 已 ready 但 daemon hello 超时”的错误码，并把 relay 慢反连窗口对齐到 20 秒，减少公网 relay 慢路径被误报成普通 `connection_closed`。
- relay device admission 增加签名、timestamp 和 nonce replay 校验；daemon 在返回 PairAccept 前确认 relay 已登记新设备，避免本地已信任但 relay 入口无法使用。
- `termd pair --qr` 的 relay pair-ticket 注册增加保守重试，公网 HTTPS 偶发超时时优先重试而不是直接返回 503。
- 安装脚本、Docker Compose 示例、部署文档和 QA 脚本已同步到 trusted relay 配对流程。

## 兼容性

- workspace / cargo / frontend 版本同步到 `0.6.0`。
- daemon、relay、Web UI 和 `termctl` 必须同步升级到 0.6.x；0.5.x 的不可信 relay/E2EE 运行时模型不能和 0.6.x trusted relay 混用。
- 旧 `relay_token` query 不再作为 trusted relay 的浏览器/termctl admission；新入口使用 setup token 注册 daemon、daemon token 认证 daemon 连接、pair-ticket/device admission 认证 client。
- supervisor 兼容版本保持 `2026-06-10`，本次升级不要求清空已有 running session supervisor。
- 现有 daemon identity 和 trusted device 本地状态可以保留；如果 relay 入口无法识别旧设备，重新执行 `termd pair --qr` 生成短期 invite 并配对。
