# v0.7 Workspace Protocol

## Authentication

The daemon Ed25519 public key is the identity anchor. Credentials use fixed-algorithm Ed25519 compact JWS with `kid=server_id`. `POST /api/auth/pair` consumes a short-lived signed pair ticket and returns a persistent device certificate. A paired device requests a nonce from `POST /api/auth/challenge`, signs the challenge with its device private key, then exchanges the proof at `POST /api/auth/access-token` for a five-minute access token. Refresh starts 60 seconds before expiry. Existing paired devices may obtain a certificate only through the restricted `POST /api/auth/device-certificate/migrate` endpoint.

The authorization schemes are `TermdPair <ticket>`, `TermdDevice <certificate>`, and `Bearer <access_token>`. A device certificate is not sufficient by itself: issuing an access token always requires proof of the matching device private key.

Direct termd and the trusted relay expose the same application contract. The relay registry stores only `server_id -> (daemon token hash, daemon public key)` and verifies signed credentials offline. It never synchronizes pair tickets, device certificates, access tokens, or device private keys. Credentials and secrets never appear in URLs, process arguments, or logs; v0.7 rejects every query credential, including `relay_token`.

## Workspace WebSockets

Each workspace owns exactly two sockets:

- `/ws/metadata` sends `metadata.snapshot` with a monotonically increasing revision, then `metadata.update` events for sessions, clients, daemon status, RTT and CWD. A revision gap closes and reconnects this socket for a fresh snapshot.
- `/ws/terminal` starts with `terminal.create` or `terminal.attach`. The server replies with `terminal.created` or `terminal.attached`, then `terminal.snapshot`, then binary supervisor terminal frames.

`terminal.snapshot` includes `session_id`, terminal size and a 1-based `cursor: { row, col }`. Subsequent cursor movement is derived from PTY output; input contains bytes rather than coordinates. Input and resize are binary terminal-socket commands. Closing an attached session closes the terminal socket immediately and sends exactly one JSON `session.close` HTTP request.

WebSocket authentication uses `Sec-WebSocket-Protocol: termd.v0.7, <access_token>`. Both routes authenticate before upgrade and reconnect with a refreshed token. The server selects only `termd.v0.7`; it never echoes the credential protocol value.

## HTTP

Control requests use `Authorization: Bearer <access_token>`, `Content-Type: application/json`, and standard JSON responses. Session list/attach/cursor/resize and daemon clients/status are not HTTP routes; their state is owned by the two sockets. HTTP `ProtocolPacket` envelopes, runtime/HTTP E2EE, session tokens/scopes, legacy download prepare/chunk routes, and fixed file polling do not exist in v0.7.

Upload chunks and download bodies are the only non-JSON application bodies. All errors, including 404 and 405, use:

```json
{"error":{"code":"not_found","message":"application route was not found","retryable":false}}
```

The retained JSON control routes cover session reorder, rename, close, legacy control noop (`POST /api/control/session/{id}/control`), search, Git, file editing and daemon client forget. File transfer uses exactly `POST /api/files/uploads`, `PUT /api/files/uploads/{id}/chunks`, `POST /api/files/uploads/{id}/commit`, `POST /api/files/uploads/{id}/abort`, `POST /api/files/downloads`, and `GET /api/files/downloads/{id}`.

## Compatibility

v0.7 is a protocol break. Application version is `0.7.0` and supervisor compatibility is `2026-07-12-dual-ws`. Upgrade relay first, then daemon. Old supervisor sessions must be cleared during the compatibility transition; daemon identity and paired-device state remain.
